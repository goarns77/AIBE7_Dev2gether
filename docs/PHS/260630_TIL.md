### Maven?
- Maven은 자바 프로젝트의 빌드, 의존성 관리, 패키징을 표준화하는 도구다.
- `pom.xml` 하나로 라이브러리, 자바 버전, 패키징 방식을 관리한다.
- 이 프로젝트도 Maven Wrapper(`mvnw`, `mvnw.cmd`)를 포함해 로컬 설치 차이를 줄였다.

```xml
<packaging>war</packaging>
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <version>6.1.0</version>
    <scope>provided</scope>
</dependency>
```

### Maven / npm
- 둘 다 패키지와 실행 흐름을 관리하는 도구다.
- `npm`이 프론트엔드/Node.js 생태계 중심이라면, Maven은 자바 생태계 중심이다.
- `package.json`이 npm의 기준 파일이라면, Maven은 `pom.xml`이 기준 파일이다.
- 차이점은 Maven이 빌드 수명주기와 산출물 규격(JAR, WAR)을 더 강하게 표준화한다는 점이다.

### pom.xml과 작성
- `pom.xml`은 프로젝트 좌표와 빌드 규칙을 적는 Maven 설정 파일이다.
- 핵심 작성 요소는 `groupId`, `artifactId`, `version`, `packaging`, `dependencies`다.
- 이 프로젝트는 Java 17, `war` 패키징, `jakarta.servlet-api`, JUnit 5를 선언했다.
- 서블릿 API를 `provided`로 둔 이유는 톰캣이 런타임에 이미 제공하기 때문이다.

### JAR / WAR
- `JAR`는 일반 자바 애플리케이션이나 라이브러리 배포 단위다.
- `WAR`는 웹 애플리케이션 배포 단위다.
- 서블릿, JSP, `WEB-INF`, 정적 리소스가 포함되는 구조라 웹 배포에는 `WAR`가 맞다.
- 현재 프로젝트가 `war` 패키징인 이유도 톰캣에 올릴 웹앱이기 때문이다.

### Jakarta EE
- Jakarta EE는 자바 기반 엔터프라이즈 웹 기술 표준 묶음이다.
- 서블릿, JSP 같은 웹 기술도 여기 포함된다.
- 예전 `javax.servlet`가 아니라 지금은 `jakarta.servlet` 패키지를 쓴다.
- `web.xml`도 Jakarta EE 6.0 스키마를 기준으로 작성되어 있다.

### 톰캣?
- 톰캣은 자바 웹 애플리케이션을 실행하는 WAS/서블릿 컨테이너다.
- 브라우저의 HTTP 요청을 받아 적절한 서블릿으로 연결하고, 응답을 다시 브라우저에 돌려준다.
- 이 프로젝트에서 톰캣은 `war`를 배포받아 `@WebServlet` 매핑과 `web.xml` 설정을 해석하는 실행 환경이다.

### 서블릿?
- 서블릿은 HTTP 요청과 응답을 처리하는 자바 클래스다.
- `HttpServlet`을 상속하고 `doGet`, `doPost`를 오버라이드해 동작을 정의한다.
- `FirstServlet`, `FormServlet`, `ScopeServlet`이 모두 이 구조를 따른다.

```java
@WebServlet("/form")
public class FormServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp)
            throws ServletException, IOException {
        req.getRequestDispatcher("form.jsp").forward(req, resp);
    }
}
```

### 라우팅 / 매핑
- 서블릿 라우팅은 URL 요청을 어떤 서블릿이 처리할지 연결하는 작업이다.
- 현재 코드는 `@WebServlet("/first")`, `@WebServlet("/form")`처럼 애너테이션으로 매핑한다.
- `web.xml`도 존재하지만, 이 프로젝트에서는 주 매핑 수단이 애너테이션이다.
- `/form` 요청이 들어오면 톰캣이 `FormServlet`으로 연결하고, 내부에서 다시 JSP로 forward한다.

### GET / POST
- GET은 조회 중심 요청이다. 주소창 접근, 페이지 진입, 목록 조회에 적합하다.
- POST는 서버 상태 변경이나 민감 정보 전달에 적합하다.
- `FormServlet`은 GET으로 `form.jsp`를 보여주고, POST로 `username`, `password`를 받는다.
- GET은 `doGet`, POST는 `doPost`로 분기된다는 점이 서블릿 기본 흐름이다.

### 서블릿+톰캣에 기반한 페이지 작성과 로그인 메소드 구현
- 흐름은 `브라우저 -> 톰캣 -> 매핑된 서블릿 -> JSP` 순서다.
- `doGet`은 페이지 진입을 담당하고, `doPost`는 사용자가 보낸 폼 데이터를 처리한다.
- 현재 `form.jsp`는 `/form`으로 POST를 보내고, `FormServlet`은 `getParameter()`로 값을 읽어 일부 마스킹 후 request attribute에 담아 다시 JSP로 넘긴다.
- 즉, 로그인 UI와 POST 처리 구조는 구현됐고, 실제 인증 로직 대신 입력값 가공 예제로 동작한다.
- 실무형 로그인 메소드는 여기서 한 단계 더 나아가 사용자 검증, 세션 저장, 실패 처리, `sendRedirect()` 기반 PRG 패턴까지 붙여야 한다.

```java
req.setAttribute("username",
        req.getParameter("username").substring(0, 3)
                + "*".repeat(req.getParameter("username").length() - 3));
req.setAttribute("password",
        req.getParameter("password").substring(0, 3)
                + "*".repeat(req.getParameter("password").length() - 3));
req.getRequestDispatcher("form.jsp").forward(req, resp);
```

### 추가 메모
- `form.jsp`의 `action="/form"`은 동작은 단순하지만, 배포 경로가 바뀌면 `contextPath`를 붙이는 편이 안전하다.
- 현재 `substring(0, 3)`은 입력 길이가 짧으면 예외가 날 수 있어 검증 코드가 필요하다.
- POST 후 바로 forward하면 새로고침 시 재전송 문제가 생길 수 있으므로 로그인/회원가입류는 PRG 패턴이 더 안전하다.
